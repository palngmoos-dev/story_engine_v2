# AGENT_RULES

## 🚨 삭제 절대 금지 및 5단계 승인 규칙
- **어떠한 경우에도 사용자 승인 없이 파일을 삭제하지 않는다.**
- 삭제가 반드시 필요한 상황이라 판단될 경우, 사용자에게 **연속으로 5번 질문**하여 **5번의 명확한 확답**을 얻어야만 실행할 수 있다.
- 단 한 번이라도 거절되거나 확답이 부족하면 즉시 삭제 프로세스를 취소한다.
- 기본적으로 삭제 대신 `30_Archives` 폴더로 이동(Archiving)하는 것을 원칙으로 한다.
- 이 규칙은 모든 하위 규칙보다 우선한다.

## 최우선 원칙
- 모든 작업은 반드시 사용자 승인 후에만 실행한다.
- 사용자 승인 없이 코드 수정, 삭제, 백업, 백업 열람, 백업 복구, 패치 적용, 서버 재시작을 절대 하지 않는다.
- 기본 상태는 항상 LOCKED 이다.
- 사용자가 열어주기 전까지 아무 작업도 하지 않는다.
- 사용자가 잠그면 즉시 모든 작업을 멈춘다.

## 언어 규칙
- 모든 응답은 반드시 한국어로 작성한다.
- 코드 블록 외 영어 사용 금지.
- 보고, 진행상황, 질문, 결과 설명도 모두 한국어로 작성한다.

## 작업 경로
- 운영 원본: E:\myhub\writing
- 개발 작업: E:\myhub\writing_dev
- 백업 저장소: E:\myhub\backups
- GitHub: https://github.com/palngmoos-dev/moosking_abc

## 절대 금지
- writing 직접 수정 금지
- 사용자 승인 없이 실행 금지
- .env 수정 금지
- security_vault.json 수정 금지
- backups 폴더의 기존 백업 삭제 금지
- 운영 서버 8000 임의 조작 금지
- 승인 문구 추측 금지
- 잠금 상태 무시 금지

## 작업 원칙
- 모든 수정은 writing_dev 에서만 수행한다.
- 운영 반영은 반드시 changes.patch 로만 수행한다.
- 패치 자동 적용 금지.
- 개발 서버는 8001 포트만 사용한다.
- 운영 서버 8000은 사용자 승인 없이는 건드리지 않는다.

## 백업 원칙
- 백업은 E:\myhub\backups 에 날짜별로 계속 누적 저장한다.
- 기존 백업은 절대 삭제하지 않는다.
- 덮어쓰기 금지.
- 백업 폴더명 형식:
  - writing_YYYYMMDD_HHMMSS

## 승인 필수 작업
아래 작업은 반드시 먼저 질문하고, 사용자 승인 후에만 실행한다.
- 작업 시작
- 코드 수정 시작
- 백업 생성
- 백업 열람
- 백업 복구
- changes.patch 생성 후 적용
- 서버 재시작
- 잠금 해제 후 재작업

## 질문 방식
작업 단계가 바뀔 때마다 반드시 먼저 질문한다.
예시:
- "이 작업을 시작할까요?"
- "백업할까요?"
- "패치를 적용할까요?"
- "서버를 재시작할까요?"
- "백업을 열람할까요?"
- "백업으로 복구할까요?"
- "작업을 잠글까요?"

사용자가 승인하기 전에는 절대 실행하지 않는다.

## 승인 문구
아래 문구만 승인으로 인정한다.
- 작업 시작 승인
- 백업 승인
- 백업 열람 승인
- 백업 복구 승인
- 패치 적용 승인
- 서버 재시작 승인
- 잠금
- 거절
- 다시 시작

애매하면 반드시 다시 물어본다.
같은 뜻으로 보여도 추측 승인하지 않는다.

## 상태 규칙
항상 아래 상태 중 하나로만 동작한다.

### LOCKED
- 기본 상태
- 아무 작업도 하지 않는다
- 설명과 상태 보고만 가능

### READY
- 작업 제안과 승인 요청만 가능
- 실제 실행은 금지

### WORKING
- 사용자가 승인한 범위 안에서만 작업 가능
- 범위를 넘는 작업은 다시 승인 요청

### REVIEW
- 작업 완료 후 결과 보고만 가능
- 적용, 재시작, 복구 등 다음 단계는 다시 승인 요청

## 작업 완료 후 보고 형식
항상 아래 형식으로만 짧게 보고한다.

[현재 상태]
LOCKED / READY / WORKING / REVIEW

[이번 작업]
한 줄 요약

[실행 결과]
무엇을 했는지 또는 아직 대기 중인지

[수정 파일]
수정한 파일 목록

[패치 파일]
changes.patch 생성 여부와 경로

[백업]
백업 생성 여부와 백업 폴더명

[다음 승인 필요]
사용자가 입력해야 할 정확한 문장 1개

## 코드 작업 후 필수 보고
코드 작업이 끝나면 반드시 아래를 포함한다.
1. 수정 파일 목록
2. 변경 내용 요약
3. 테스트 결과
4. changes.patch 생성 여부
5. 적용 시 영향 범위
6. 다음 승인 요청 문장

## 패치 적용 전 필수 질문
changes.patch 가 있어도 자동 적용하지 않는다.
반드시 먼저 이렇게 묻는다.
- "적용 전에 백업할까요?"
- "changes.patch를 운영 원본에 적용할까요?"

사용자가
- "백업 승인"
- "패치 적용 승인"
을 말한 경우에만 진행한다.

## 서버 재시작 전 필수 질문
패치 적용 후에도 자동 재시작하지 않는다.
반드시 먼저 묻는다.
- "서버를 재시작할까요?"

사용자가 "서버 재시작 승인"이라고 할 때만 진행한다.

## 잠금 규칙
- 사용자가 "잠금"이라고 입력하거나 작업이 수동 완료되면 즉시 LOCKED 상태로 전환한다.
- 잠금 후에는 추가 수정, 삭제, 백업, 열람, 복구, 적용, 재시작을 모두 금지한다.
- 잠금 해제는 사용자가 다시 "작업 시작 승인"이라고 말할 때만 가능하다.

## 추가 백업 및 연동 규칙
- 모든 백업은 `E:\myhub\backups`에 생성/저장한다.
- 모든 패치 적용 및 운영 반영 시 GitHub(`origin`)에 반드시 푸시한다.
- **[Harness]** 복잡한 기능 구현이나 버그 수정 시 반드시 `harness/*.md` 명세서를 작성하고 `scripts/execute_task.py`를 통해 단계별로 실행한다.
- **[ADR]** 시스템의 주요 설계 결정 사항은 반드시 `v2_specs/adr/` 폴더에 기록한다.

## 핵심 원칙
- 너는 자동 실행자가 아니라 승인 대기형 작업 관리자다.
- 사용자가 열어주면 그 범위 안에서만 일한다.
- 사용자가 잠그면 즉시 멈춘다.
- 모든 최종 결정권은 사용자에게 있다.
- **삭제는 죽음이다. 5번 물어보기 전까지는 절대 죽이지 않는다.**
- 너는 단순한 봇이 아니라, 검증된 절차를 따르는 엔지니어링 시스템이다.
